Hello,

I was measuring areas limited to threshold on 16bit image stacks (to imagine
the data: circular gradients, white on black), updated version today.  I
realized that the measured area was far smaller than the one highlighted red in
the image. 
I found out that the highest possible value of the min-max-sliders of the
threshold-dialogue-box does not belong to the actual image (and also not to the
actual stack). It changes to proper values only when i use "stack histogram"
(or after that un-select "stack histogram" to get the values for the actual
slice). Closing and re-opening the threshold-dialogue-box also does not help.
The values also are not actualized when setting a new minimum/maximum value via
the sliders (using "set" updates them). However, changing the maximum slider
(just by one step) updates the look-up table, so not the entire area is
highlighted red.
The problem may  occur because of changing between different image stacks. (I
was clicking around now a bit and it happens quite sometimes, but i couldn't
find out when exactly.)
I don't mind the values stay the same, this is just inconvenient -updating the
stack histogram fixes it. But i think it is a big problem that the red
highlighted region is not the thresholded region. If the the wrong value is not
too far from the true one, it is hard for the user to realize that not the
dispayed region is evaluated.

Best regards & big thanks for developing this software,
Felix

Hi,

I have recently come across a problem in the "Analyse Particles" function which
I think might be a bug in handling of hyperstacks or ROI position handling. 

I analyse a thresholded timeseries (Z + T) and add identified particles to the
ROI manager. I should note that I get the time series by the "Duplicate"
command from a composite image.

At the first timepoint, ROIs are added to the ROI manager for each for each of
the Z-Planes. 
However, problems start to occur once the stack reaches the last z-plane of
timepoint 1. I would expect that now the ROIs from timepoint 2 should be added
to the manager. 

However, when reaching the end of the Z-Stack of timepoint 1, the ROI positions
jump not to timepoint 2, but rather to the next timepoint that would come after
the Z-Stack, e.g. if the z-stack has 5 planes, the next added ROIs will be
localized to Timepoint 6, Z-Position 5.  After this, the ROI-Positions will
cycle not through the Z-planes, but rather through the time-axis until the end
is reached. After this, the time-axis jumps to a later timepoint (in the case
of the "Mitosis 5D" sample timeseries to timepoint 11) and from there on
behaves as expected, e.g. ROIs are added first to z-planes and when the
last-Z-plane is reached, ROIs will be added to the first Z-plane of the next
timepoint..
I do not think that the problem is with the definition of the ROIs as such,
since the shape of the ROIs corresponds to the segmented particles at the
expected timepoints. I rather think that wrong position info is written to the
ROIs when adding them to the ROI manager. 
The problem occurs both in FIJI and in plain ImageJ, both in a current version
and in the FIJI lifeline version.
I use Windows 7 64bit version, and run ImageJ / Fiji in 64 bit. However, I have
tested the plain ImageJ 32 bit version and find the same behaviour.

extracts the first channel, applies a threshold  and then runs the Particle
analyser which adds the ROIs to the manager. Then the positions of the ROIs in

Kay

--- Comment #6 from Wayne Rasband <wsr at nih.gov> ---
It appears you have two displays, both with a resolution of 1920x1080. ImageJ
will show dialogs on the display on the left ("Monitor1") because it has an
origin of (0,0). You might be able to work around this problem by swapping the
two displays so that the primary display is the one on the left. ImageJ cannot
use the display on the right ("Monitor2") because it's origin is not at (0,0),
which is normally the case on Linux. There is a lot of code in ImageJ, and in
plugins and macros, that assume that the primary display has its origin at
(0,0).

--- Comment #1 from Jan Eglinger <jan.eglinger at uni-duesseldorf.de> ---
Mike, did you follow the instructions here:
http://fiji.sc/Install_Fiji_on_MacOSX
to install Fiji?

On Mac OSX, applications are installed by copying them from the mounted .dmg
file (which is just a virtual installation disk) to the Applications folder or
any other local folder with read-write access. (You can then eject the mounted
disk image and move the .dmg file to the trash.)
For Fiji, it is recommended to copy the application to your Desktop or your
user's home folder to avoid problems on multi-user systems.

Hope that helps.
